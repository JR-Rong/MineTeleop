# 项目背景与已确认决策

## 目标

建设一套矿车遥操系统，分别运行在：

- 车端：Ubuntu 工控机，连接相机和车辆控制接口。
- 驾驶端：远端模拟驾驶器，优先 Windows 原生桌面程序，也兼容 Linux。
- 云端：固定 IP 服务器或独立实时节点，用于信令、鉴权、NAT 穿透、TURN 兜底和录像上传协调。

系统需要在车端实时发送多路图像到驾驶端，驾驶端发送档位、转向、油门、刹车等控制命令到车端。车端还需要保存原分辨率录像，并按策略逐文件上传云端；视频文件本身已编码压缩，不把 zip/tar 打包作为默认省流量手段。

## 使用场景

- 场地：封闭矿区。
- 车辆：矿车。
- 车速：最高不超过 40 km/h。
- 网络：车端与驾驶端不在同一个局域网，车端使用 5G。
- 目标：低延迟优先，同时尽可能降低 5G 流量。

该场景不是公开道路首版部署，因此第一版不引入公开道路法规、复杂调度和多驾驶员抢占。但由于车辆真实运行，失联安全停车、权限控制和审计日志必须从第一版开始设计。

## 已确认需求

- 首版会话模型：一车一驾驶员。
- 驾驶端形态：原生桌面程序。
- 驾驶端输入：第一版使用键盘/软件控件，预留方向盘/踏板/HID/CAN/串口适配。
- 相机输入：USB/CSI/GMSL 原始相机优先。
- 相机数量：默认 4 路，可新增或减少。
- 实时视频默认：720p 30 fps，可配置。
- 录像质量：按采集原分辨率编码保存，不与实时流强绑定。
- 控制频率：20 Hz。
- 失联策略：控制链路异常时车端安全停车。
- 云端中继策略：云端做信令和中继兜底，媒体优先走最低延迟路径。
- 云端 UDP：可以新增支持 UDP 的节点。

## 关键技术结论

### 传输

推荐以 WebRTC 为实时媒体和控制通道主线：

- 媒体：多路低延迟视频。
- 控制：优先使用 WebRTC DataChannel 的 unordered/unreliable 模式；如果改用独立通道，也必须保持低延迟、丢弃旧命令和只执行最新有效状态的语义。
- 穿透：STUN/TURN。
- 信令：HTTPS/WebSocket。

WebRTC 的优势是 NAT 场景成熟、低延迟、拥塞控制和丢包恢复能力较好，并且可以用 TURN 兜底。对于 50-150 ms 目标，云中继节点必须支持 UDP，且最好与现有业务隔离。

### 编码

当前工控机硬件排查结论：

- 没有检测到 NVIDIA 设备或可用 NVIDIA 驱动。
- 检测到 Intel Alder Lake-S GT1 核显。
- `/dev/dri/renderD128` 存在。
- Docker 内 Ubuntu 22.04 环境已验证 `h264_vaapi` 编码成功。
- 宿主机 `/usr/local/bin/ffmpeg` 与系统 `libva2` ABI 不匹配，不能作为硬件不可用的证据。
- 宿主机 apt/ROS 依赖处于 broken 状态，短期不建议为了媒体验证强修宿主机 apt。

因此，项目编码主线应设为：

1. Intel VAAPI/QSV 优先。
2. CPU x264 兜底。
3. 不按 NVIDIA NVENC 作为当前工控机默认能力设计。
4. 运行环境尽量容器化，固定 `ffmpeg/GStreamer/libva/intel-media-driver` 版本。

## 重要设计原则

- 实时链路和录像上传必须解耦。
- 实时视频宁可丢帧，也不能无限排队导致驾驶画面变旧。
- 控制命令必须有协议版本、序号、时间戳和心跳；车端以序号和本地到达间隔判断新鲜度，跨机器时间戳主要用于审计和追溯。
- 控制链路异常时，车端本地状态机必须独立执行分级减速安全停车，不能依赖云端或驾驶端持续发包维持安全动作。
- 控制接收、安全状态机和车辆适配器必须与媒体编码 pipeline 做故障隔离，避免 GStreamer/VAAPI 卡死时拖死安全逻辑。
- 车端、驾驶端和云端必须有明确的 NTP 时间同步要求；如后续需要多相机严格同步或更高精度控制追溯，再评估 PTP。
- 所有关键参数可配置，并能在日志中追溯实际生效值。
- 第一版只做一车一驾驶员，控制权简单明确。
- 对未知硬件接口使用 Adapter 预留，不提前绑定 CAN 或动态库。
