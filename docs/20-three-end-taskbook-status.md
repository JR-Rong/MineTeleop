# Three-end taskbook delivery status

This matrix is the authoritative implementation checklist for the vehicle,
control, and server taskbook. A source file or unit test is not treated as
field acceptance. Status values are:

- **implemented**: current code and an automated check cover the stated item.
- **partial**: useful code exists, but one or more required paths or proofs are
  missing.
- **not implemented**: the current architecture does not yet provide the
  required behavior.
- **field gate**: target hardware, public networking, or physical vehicle proof
  is mandatory.

## Joint acceptance

All joint gates remain **not accepted** until evidence is produced for two
vehicles, two controls, public TLS/STUN/TURN paths, two simultaneous camera
tracks per vehicle, the 20 Hz DataChannel, P95 latency, 30-minute and 8-hour
soaks, three control OS packages, and real vehicle CAN/braking safety.

The narrower Ubuntu vehicle + cloud + Mac delivery is accepted for one vehicle:
two physical camera tracks, zero-state DataChannel control, a 30-minute soak,
safe logout, 5 ms cross-host time uncertainty, direct signaling without
SSH/SOCKS/FRP, and vehicle re-registration after a signaling restart all have
live evidence. That result does not satisfy the broader two-vehicle,
three-control-OS, forced-TURN, 8-hour, per-camera-failure, or physical-braking
gates above.

The repository must never use a unit test, local loopback, Docker control-plane
smoke, or simulator run as a substitute for those field artifacts.
